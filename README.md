# Тестовая задача в виде модуля.

Для запуска:

* php 8.3
* Устанавливаем б24 через установочный скрипт https://www.1c-bitrix.ru/download/cms.php#tab-subsection-2
* Добавляем файлы из репозитория
* В корень модуля test добавляем файл .env с содержимым.
* COURIER_MOCK=http://b24.loc/courierQuoteMock.php (b24.loc меняем на свой домен)
* Накатываем бекап.
* После всех проделанных манипуляций рекомендовано настроить Главный модуль и указать домен.

Для запуска тестов:
* composer update
* vendor\bin\phpunit tests/unit/modules/test/AppTest.php

## Отклонения от ТЗ
Внешний сервис http://courier-mock:8080/v1/quote заменен на внутренний http://b24.loc/courierQuoteMock.php в целях
более быстрой реализации. В реальных условиях для реализации был описан отдельный класс АПИ который взаимодействует
с внешним сервисом и методами подходящими по условия внешнего сервиса.
Из минусов не удалось продемонстрировать идемпотентность, и работу с лимитами в рамках тестовой задачи.

Пункт 8 реализован без использования REST т.к. на пробной версии б24 REST не доступен.

## Описание работы модуля и комментарии разработчика

Весь функционал в описан модуле, а для работы нужен лишь один метод process, который запускается в файле init.php при обновлении сделки. 
Что бы расчет прошел успешно в сделке необходимо выбрать регион доставки, иначе расчет не произойдет. Регионы добавляются в админке через инфоблок (Регионы доставки)
При добавлении значения инфоблока важно заполнять поле "CODE" т.к. по нему идет сопоставление в функции courierQuoteMock.

В качестве аргумента метода process лучше было бы принимать массив сделки который передает битрикс при наступлении события обновления сделки.
Это бы избавило от некоторых избыточных вычислений которые происходят в вызываемом методе.

Настройки с названием стадий и метода лучше было бы перенести в настройки модуля т.к. при установке на новый портал все ИД указанные в классе будут бесполезны.

В качестве trace-id используется ИД сделки, который заполняется только в случае успешного расчета, что достаточно для избежания ретраев расчета при повторной смене стадии.

В случае ошибки при расчете ошибка логируется в папке модуля. Возможно стоило сделать запись в таймлайн, что бы сотрудник работающий со сделкой мог понимать проблему.

Тесты текущие реализованны для галочки. Можно было протестировать значения в инфоблоке с регионами (на случай ввода некорректного кода региона). (времени уже нет)

В сервис courierQuoteMock.php можно было добавить авторизацию и рандомную выдачу результата с разными ответами сервиса включая 503 ошибку при слишком частых запросах. (времени уже нет)
